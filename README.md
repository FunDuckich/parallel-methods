## Подробные пояснения по MPI-заданиям

### Что такое MPI
MPI — это стандарт для параллельных программ, где несколько процессов обмениваются сообщениями. В отличие от OpenMP, где внутри одной программы создаются потоки, MPI запускает несколько отдельных процессов. Каждый процесс выполняет один и тот же `.exe`, но имеет свой номер.

`MPI_Init(&argc, &argv)` запускает MPI-среду. `argc` и `argv` передаются по адресу, потому что MPI может прочитать и обработать служебные аргументы запуска.

`MPI_COMM_WORLD` — общий коммуникатор, в который входят все процессы, запущенные через `mpiexec`.

`MPI_Comm_rank(MPI_COMM_WORLD, &id)` записывает в `id` номер текущего процесса.

`MPI_Comm_size(MPI_COMM_WORLD, &count)` записывает в `count` общее число процессов.

`MPI_Finalize()` завершает работу MPI и освобождает ресурсы. После него MPI-функции уже не вызывают.

### Лаба 6, задание 1
Смысл задания — познакомиться с базовой структурой MPI-программы. Нужно, чтобы каждый процесс вывел свой номер и общее количество процессов.

Если программа запускается командой `mpiexec -n 4`, MPI создаёт 4 процесса. Каждый процесс вызывает `MPI_Comm_rank` и получает свой номер: `0`, `1`, `2`, `3`. Потом каждый печатает строку вида `process 2 of 4`.

Процесс `0` дополнительно выводит сравнение интерфейсов MPI для C и C++. В C используется стиль функций: `MPI_Comm_rank(MPI_COMM_WORLD, &id)`. В старых C++ bindings использовался объектный стиль: `MPI::COMM_WORLD.Get_rank()`. В современных программах на C++ обычно всё равно используют C-интерфейс MPI, потому что старые C++ bindings считаются устаревшими.

### Лаба 6, задание 3
Смысл задания — отправить вещественное число двойной точности от процесса `0` процессу `1` с буферизацией.

Обычная отправка `MPI_Send` может ждать, пока принимающий процесс будет готов принять сообщение. Буферизованная отправка `MPI_Bsend` сначала копирует данные в заранее подключённый буфер. После этого процесс-отправитель может продолжить выполнение.

`double value = 12.75` — число, которое отправляется.

`char buffer[MPI_BSEND_OVERHEAD + sizeof(double)]` — область памяти для буферизованной отправки. Используется `char`, потому что `char` занимает 1 байт, а буфер — это просто набор байтов. `sizeof(double)` даёт размер самого числа, а `MPI_BSEND_OVERHEAD` добавляет место для служебной информации MPI.

`MPI_Buffer_attach(buffer, buffer_size)` подключает буфер к MPI.

`MPI_Bsend(&value, 1, MPI_DOUBLE, 1, 0, MPI_COMM_WORLD)` отправляет одно число типа `double` процессу `1`. Последний `0` перед коммуникатором — тег сообщения. Тег нужен, чтобы различать разные типы сообщений между одними и теми же процессами.

`MPI_Recv(&value, 1, MPI_DOUBLE, 0, 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE)` принимает одно число типа `double` от процесса `0` с тегом `0`. `MPI_STATUS_IGNORE` означает, что дополнительная информация о сообщении нам не нужна.

### Лаба 6, задание 6
Смысл задания — принять сообщение заранее неизвестной длины. Проблема в том, что для `MPI_Recv` обычно нужно заранее знать размер буфера, куда будут записаны данные.

Процесс `0` отправляет строку:

```cpp
message with unknown length
```

Отправляется `strlen(message) + 1`, потому что вместе с символами строки передаётся завершающий нулевой символ `\0`.

Процесс `1` сначала вызывает `MPI_Probe(0, 0, MPI_COMM_WORLD, &status)`. Эта функция ждёт сообщение от процесса `0` с тегом `0`, но ещё не забирает его. Она только заполняет `status` информацией о найденном сообщении.

`MPI_Get_count(&status, MPI_CHAR, &length)` узнаёт, сколько элементов типа `MPI_CHAR` находится в этом сообщении. После этого можно выделить память точного размера: `new char[length]`.

Затем обычный `MPI_Recv` принимает уже само сообщение. Такое решение нужно, когда размер данных заранее неизвестен.

### Лаба 7, задание 1
Смысл задания — измерить две характеристики коммуникационной среды: латентность и пропускную способность.

Латентность — это минимальное время передачи сообщения. Для её оценки используется сообщение нулевой длины. Оно почти не содержит данных, поэтому время в основном показывает накладные расходы на сам факт обмена.

Пропускная способность — это количество данных, которое можно передать за секунду. В программе проверяются сообщения разных размеров: от `0` до нескольких мегабайт. Для каждого размера выполняется много повторов по схеме ping-pong: процесс `0` отправляет сообщение процессу `1`, а процесс `1` отправляет его обратно.

Время одного направления считается так:

```text
one_way_time = full_time / (2 * repeats)
```

Деление на `2` нужно, потому что один повтор состоит из отправки туда и обратно.

Пропускная способность считается так:

```text
bandwidth = message_size / one_way_time
```

В коде результат переводится в `MB/s`. Программа запоминает максимальную достигнутую пропускную способность и размер сообщения, на котором она получилась.

### Лаба 7, задание 3
Смысл задания — организовать обмен между соседними процессами по кольцу с использованием неблокирующих операций отправки и приёма.

Кольцо означает, что каждый процесс отправляет данные следующему процессу, а принимает от предыдущего. Для процесса `id` следующий вычисляется так:

```cpp
next = (id + 1) % count
```

Предыдущий:

```cpp
prev = (id - 1 + count) % count
```

Операция `% count` замыкает цепочку в кольцо: последний процесс отправляет процессу `0`, а процесс `0` принимает от последнего.

`MPI_Irecv` запускает неблокирующий приём. Он не ждёт завершения сразу, а возвращает объект запроса `MPI_Request`.

`MPI_Isend` запускает неблокирующую отправку. Она тоже не обязана завершиться сразу.

`MPI_Waitall(2, requests, MPI_STATUSES_IGNORE)` ждёт завершения сразу двух операций: приёма и отправки. Такой вариант удобен для кольца, потому что все процессы сначала размещают приём и отправку, а потом ждут завершения. Это снижает риск взаимной блокировки.

### Лаба 7, задание 6
Смысл задания — показать схему master-slave. Процесс `0` является ведущим, остальные процессы — ведомыми.

Ведомые процессы считают простое значение `id * id` и отправляют его мастеру через блокирующий `MPI_Send`.

Мастер заранее создаёт неблокирующие приёмы от всех ведомых:

```cpp
MPI_Irecv(&values[i], 1, MPI_INT, i + 1, 0, MPI_COMM_WORLD, &requests[i])
```

После этого он вызывает `MPI_Waitsome`. Эта функция ждёт, пока завершится хотя бы часть запросов. То есть мастеру не нужно принимать сообщения строго по порядку. Если процесс `3` прислал раньше процесса `1`, мастер может обработать сообщение от `3` сразу.

В `indexes` попадают номера завершившихся запросов, а в `statuses` — информация о сообщениях, например `MPI_SOURCE`, то есть номер процесса-отправителя.

Такое решение подходит для master-slave схем, где ведомые могут завершать работу в разное время.

### Лаба 8, задание 1
Смысл задания — вычислить матрицу `A`, затем получить матрицу `B = A^T`, то есть транспонированную матрицу. Транспонирование означает, что строки становятся столбцами:

```text
B[j][i] = A[i][j]
```

Элемент матрицы `A` вычисляется по формуле:

```text
a[m][n] = sum from k = 1 to N of (sin(m + k) + cos(n - 2k)) / (1 + (m - n)^2 + k^2)
```

В программе используется `N = 6`, чтобы вывод матрицы был небольшим и читаемым.

Решение построено по схеме master-slave. Процесс `0` — ведущий. Он не считает строки сам, а распределяет работу между ведомыми процессами. Каждый ведомый процесс получает свой диапазон строк матрицы `A`, вычисляет их и отправляет мастеру.

Мастер заранее ставит неблокирующие приёмы через `MPI_Irecv`. Затем он использует `MPI_Waitany`, чтобы дождаться любого готового блока строк. Это удобно, потому что ведомые процессы могут закончить работу в разном порядке.

Когда мастер получает строки, он сразу выполняет транспонирование:

```cpp
B[j][row] = rows[row * N + j]
```

Итог — матрица `B`, которая является транспонированной версией `A`.

### Лаба 8, задание 3
Смысл задания — ускорить вычисление интеграла по квадратурной формуле Симпсона с помощью MPI и операции редукции.

В задании проверяется тождество:

```text
integral from 0 to infinity of 1 / (1 + x^2) dx = pi / 2
```

Но считать бесконечный промежуток напрямую неудобно. Поэтому используется замена переменной, которая переводит промежуток `[0, infinity)` в `[0, 1]`. После замены получается функция:

```text
f(x) = 1 / (x^2 + (1 - x)^2)
```

Её интеграл на `[0, 1]` тоже равен `pi / 2`.

Формула Симпсона использует значения функции в нечётных и чётных внутренних точках:

```text
integral = h / 3 * (f(a) + f(b) + 4 * sum1 + 2 * sum2)
```

`sum1` — сумма значений в точках `a + (2k - 1)h`.

`sum2` — сумма значений в точках `a + 2kh`.

Каждый MPI-процесс считает только часть слагаемых: `k = id + 1`, потом `k += count`. Так работа равномерно распределяется между процессами.

`MPI_Reduce` собирает локальные суммы всех процессов в одну общую сумму на процессе `0`. Используется операция `MPI_SUM`, то есть все частичные суммы складываются.

После этого процесс `0` считает итоговый интеграл и сравнивает его с `acos(-1.0) / 2.0`, то есть с `pi / 2`.

### Лаба 8, задание 7
Смысл задания — создать пользовательский MPI-тип данных для структуры частицы и передать массив таких структур.

Обычные типы MPI вроде `MPI_INT`, `MPI_DOUBLE`, `MPI_CHAR` подходят для простых данных. Но структура `Particle` содержит разные поля:

```cpp
double x, y, z;
int charge;
double mass;
char extra[8];
```

Такую структуру нельзя корректно передать как простой массив байтов без учёта выравнивания памяти. Компилятор может вставлять пустые байты между полями, поэтому MPI нужно объяснить расположение каждого поля.

`block_lengths` задаёт количество элементов в каждом блоке: три координаты `double`, один заряд `int`, одна масса `double`, восемь символов `char`.

`offsets` задаёт смещения полей внутри структуры. Для этого используется `offsetof`, чтобы не считать смещения вручную.

`types` задаёт MPI-типы для каждого блока.

`MPI_Type_create_struct` создаёт новый тип из этих блоков.

`MPI_Type_create_resized` задаёт правильный шаг между элементами массива структур. Это важно именно для передачи массива `Particle particles[3]`, чтобы MPI понимал, где начинается следующая структура.

`MPI_Type_commit` регистрирует тип, после чего его можно использовать в `MPI_Send` и `MPI_Recv`.

Процесс `0` заполняет массив из трёх частиц и отправляет его процессу `1`. Процесс `1` принимает массив и печатает все поля. В конце `MPI_Type_free` освобождает созданные MPI-типы.
